## Product Goal

Build a static GitHub Pages voter guide for the 2026 South Korean local elections.

Core experience:

- A voter selects where they live.
- The app shows only the ballot groups and candidates that voter is expected to see.
- Candidate cards show party, photo, pledge summary, comparison points, profile-specific relevance, and official-data-only notices.
- Candidate detail dialog shows full pledges, fact checks, persona prompt, and source boundaries.
- No backend and no database. Data is generated into static JSON at build time.

Important product constraints:

- Do not recommend a candidate.
- Do not claim political neutrality by assertion only; show source and prompt boundaries.
- Use NEC and official public information conservatively.
- Hallucination risk must be treated as a product bug. If evidence is insufficient, say so.
- Keep implementation/cache details out of voter-facing UI.

## Static Data Architecture

No DB is intended.

Build pipeline:

- `npm run data:build`
- `npm run build` runs `data:build`, TypeScript, then Vite.
- `scripts/build-data.ts` builds region shards and share pages from committed/static data plus generated NEC cache.

Key generated outputs:

- `public/data/regions/nec-*.json`
- `public/share/*.html`
- `public/data/cache-manifest.json`
- `public/data/regions/index.json`

Current `.gitignore` intentionally ignores:

- `public/data/nec/`
- `public/data/regions/nec-*.json`
- `public/share/`

Result:

- GitHub Pages can still deploy full nationwide static output because the workflow runs `npm run build`.
- A fresh local checkout may not have all generated region shards until `npm run data:build` or `npm run build` is run.
- Be careful not to commit huge generated shards unless the deployment strategy changes.

Known local data size from previous inspection:

- `public/data` around 476 MB locally due ignored generated data.
- `dist/data` around 568 MB after build.
- `data/nec/full` around 163 MB.
